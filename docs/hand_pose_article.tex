\noindent\textbf{\textit{Abstract}}---This article presents an initial research-article draft for a project on lightweight hand pose tracking for real-time human-computer interaction on edge devices. The longer-term project aim is to study both efficient RGB and stereo-aware pipelines for touchless interaction, but the current repository results are limited to a DRHand-inspired two-dimensional RGB estimator, its ablation experiments on RHD, and its deployment benchmarks on an NVIDIA Jetson Orin Nano. The implemented system combines a lightweight backbone with heatmap and coordinate regression branches together with a geometric fusion rule. The current experiments compare 21-joint and 10-joint variants, isolate the contribution of fusion against heatmap-only and coordinate-only outputs, and report both predictive accuracy and end-to-end latency. These results provide an initial picture of the present RGB pipeline and establish a baseline for later comparison with broader evaluation, including stereo input, application-level interaction testing, and further model refinement.

\noindent\textbf{\textit{Keywords}}---hand pose estimation, human-computer interaction, edge devices, lightweight deep learning, stereo vision

\section{INTRODUCTION}

Human-computer interaction increasingly extends beyond keyboard, mouse, and touchscreen input toward camera-based, touchless interfaces. Hand pose tracking is attractive in this setting because hands are the main instrument for pointing, manipulation, and expressive gesture, and prior surveys identify hand-based interaction as especially promising for contact-free control, public displays, and sterile environments \cite{vuletic,wachs,li_vr}. Recent systems such as MediaPipe Hands show that near-real-time landmark prediction from RGB input is practical on commodity devices \cite{mediapipe}, which makes hand pose a plausible interface primitive for everyday HCI rather than only a research benchmark.

Despite that promise, reliable hand pose estimation remains hard under self-occlusion, cluttered backgrounds, fast motion, challenging viewpoints, and limited compute budgets. These problems become more severe on edge devices, where latency, memory, power, and thermal limits constrain model choice. Lightweight 2D pipelines such as DRHand, SRHandNet, FastHand, and other efficiency-focused architectures address part of this problem \cite{dualreg,srhandnet,fasthand,attentionlite}, but accuracy-latency trade-offs remain central when a model must support interactive response rather than offline analysis.

Stereo or depth-aware input may improve robustness to distance ambiguity and occlusion, yet practical embedded deployments remain less explored for detailed hand tracking than for broader pose estimation tasks \cite{stereojetson,blazepose}. The wider project that motivates this article therefore targets lightweight hand pose tracking for real-time HCI on embedded NVIDIA hardware, with future comparison between RGB-only and stereo-aware pipelines. The current repository captures the first concrete stage of that agenda: a reproducible RGB 2D hand-pose workflow with training, evaluation, MediaPipe comparison support, and Jetson-oriented benchmarking.

This article reports the current state of that workflow using the aggregated results already present in the repository. As an initial draft, its purpose is to document the present system, summarize the current experiments, and provide a structured basis for later revision. The article therefore focuses on the existing RGB experiments, the current benchmark setup, and the model behaviour visible from the available result tables.

\section{LITERATURE REVIEW}

Related work spans both interaction design and pose estimation. Surveys of gesture-based HCI argue that hand input can be natural and expressive, particularly in virtual and mixed reality, large-display control, and clinical settings where touchless operation is valuable \cite{vuletic,wachs,li_vr}. On-device body-pose systems such as BlazePose further demonstrate how mobile deployment changes model priorities by rewarding low latency and stable tracking over absolute leaderboard performance \cite{blazepose}.

Within hand pose estimation itself, MediaPipe Hands uses a detector-plus-landmark pipeline aimed directly at on-device deployment \cite{mediapipe}. SRHandNet, FastHand, and Attention! each pursue lightweight inference through efficiency-focused architectural choices \cite{srhandnet,fasthand,attentionlite}. DRHand is especially relevant to the present repository because it combines a lightweight backbone with both heatmap and coordinate regression branches, then fuses their outputs geometrically \cite{dualreg}. The repository studied here adopts that same high-level idea and packages it as a modular experimental codebase.

Other methods emphasize structure and generalization. NSRM introduces nonparametric structure regularization for 2D hand pose \cite{nsrm}, while depthwise separable convolution and related efficient backbone design remain common tools for reducing computation \cite{xception}. The coordinate branch in the current repository uses Wing loss, which is designed for precise landmark localization when small residual differences matter \cite{wingloss}. For evaluation, the experiments rely on the RHD benchmark introduced by Zimmermann and Brox, which remains a useful testbed because it contains diverse synthetic poses, viewpoints, and occlusion patterns \cite{rhd}.

Stereo, RGB-D, and egocentric settings highlight the next challenge. Stereo systems on embedded NVIDIA hardware have been demonstrated for real-time pose assessment, but such work typically targets whole-body tasks or generic pose estimation rather than lightweight, fine-grained hand tracking for HCI \cite{stereojetson}. This leaves a practical gap between efficient single-RGB hand tracking and richer multi-view sensing, and it motivates the later stages of the project described in the preliminary article.

\section{TECHNICAL APPROACH}

\subsection{Project Scope and Deployment Context}

The broader project aim is real-time touchless HCI on edge devices, potentially using both monocular RGB and stereo RGB input. The current repository and aggregated results correspond only to the RGB 2D stage. No stereo fusion or ZED-specific training pipeline is presently evaluated in the reported tables. This section therefore documents the currently implemented RGB approach rather than the full planned system.

\subsection{Repository Design}

The repository is organized around a small Python package under \texttt{src/handpose} and a script layer under \texttt{scripts}. Data loading, model definitions, training utilities, evaluation code, and inference helpers are separated into modules so that experiments can be reproduced from command-line entry points. In addition to the DRHand-style model, the codebase includes a MediaPipe evaluation script and an end-to-end benchmark pipeline intended for Jetson-class deployment. The training script follows a fixed two-stage procedure. Stage 1 optimizes the backbone and heatmap head only. Stage 2 adds the coordinate head and optionally freezes earlier components.

The implemented network contains three parts: a shared backbone, a heatmap regression head, and a coordinate regression head. The backbone outputs an $8 \times 8 \times 288$ feature tensor. The heatmap head upsamples this representation to per-joint heatmaps, while the coordinate head regresses normalized $(x,y)$ positions from intermediate features. This organization mirrors the architecture described in DRHand and preserves the intended contrast between dense spatial supervision and direct coordinate prediction \cite{dualreg}.

\subsection{Datasets and Preprocessing}

The repository supports RHD and a COCO-style hand-keypoint dataset through a shared dataset factory. The current article reports only the ablation results available for RHD because the aggregated result files in \texttt{docs/aggregated\_results} are centered on that benchmark. RHD is especially suitable for this analysis because it is challenging, multi-pose, and contains visibility annotations that allow metrics to ignore occluded joints when appropriate \cite{rhd}. At this stage, the evaluation remains benchmark-centric rather than task-centric: no user-study or application-loop dataset is yet part of the reported pipeline.

All reported experiments use full-image resize preprocessing rather than detector-based cropping. Each RGB frame is resized to $256 \times 256$, and all target coordinates are rescaled into normalized image coordinates. This design simplifies the pipeline and makes training, evaluation, and benchmarking consistent, but it also means that the reported results are conditioned on full-frame hand localization rather than an external detection stage.

\subsection{Training Objective and Fusion Rule}

Stage 1 uses mean-squared error between predicted heatmaps and Gaussian targets generated from normalized keypoint coordinates. Stage 2 combines the heatmap loss with Wing loss on the coordinate branch \cite{wingloss}. The repository supports either the full 21-joint layout or a reduced 10-joint layout composed of finger bases and tips.

Inference exposes three prediction modes: \texttt{heatmap}, \texttt{coord}, and \texttt{fusion}. Heatmap mode applies argmax decoding to each predicted heatmap. Coordinate mode returns the direct regression output. Fusion mode compares the disagreement between the two branches for each joint to a threshold $\alpha$, where $\alpha$ is the median predicted bone length in the current hand instance. Heatmap coordinates are selected when the disagreement is smaller than $\alpha$; otherwise coordinate predictions are retained. This rule follows the design rationale of DRHand and aims to combine local precision with better structural stability \cite{dualreg}.

\subsection{Evaluation and Aggregation}

The evaluation pipeline reports three principal metrics: normalized sum of squared error (SSE), root-relative normalized end-point error (EPE), and PCK@0.2. Benchmarking is performed separately on Jetson Orin Nano and reports per-image latency for image read, preprocessing, host-to-device transfer, forward prediction, JSON serialization, and end-to-end runtime. These measurements document the current experimental conditions, but the repository does not yet integrate task-completion or user-study measures. The tables in this article are populated from the aggregated CSV outputs already stored in \texttt{docs/aggregated\_results}. Each table entry is therefore presented as mean $\pm$ standard deviation over five seeds.

\section{RESULTS AND ANALYSIS}

The reported results correspond to the current RGB-only stage of the project. They should be treated as preliminary results for the present implementation rather than as final conclusions about the full project. Unless otherwise stated, the accuracy tables summarize five-seed experiments on RHD and the latency tables summarize Jetson Orin Nano benchmarks exported by the repository benchmark pipeline.

\subsection{Main Accuracy Results}

Table~\ref{tab:main_accuracy} compares the 21-joint baseline (\texttt{B0}) and the reduced 10-joint model (\texttt{B1}). Native evaluation should be interpreted carefully because the models predict different numbers of keypoints. Under native evaluation, the reduced model shows lower SSE and EPE largely because it solves a smaller prediction problem. The fairer comparison is the shared-10 protocol, where the 21-joint checkpoint is evaluated only on the 10 keypoints present in the reduced model. Under this setting, both models are extremely close: \texttt{B0} gives slightly lower SSE, whereas \texttt{B1} gives slightly lower EPE, and the PCK values are almost identical.

This comparison shows that the reduced keypoint layout retains most of the predictive behaviour for the shared subset while reducing the size of the prediction task. It also indicates that native 21-joint and native 10-joint metrics should not be compared directly without considering the number of evaluated points.

\subsection{Branch Ablation}

Table~\ref{tab:branch_ablation} isolates the behavior of the three output modes. A consistent pattern emerges. Heatmap decoding gives the highest PCK for the 21-joint baseline, which suggests that it often places points inside the acceptance radius for PCK. However, its EPE is clearly worse than either fusion or coordinate regression, which indicates that PCK alone is not sufficient to characterize geometric quality. Fusion obtains the best SSE and EPE for both \texttt{B0} and \texttt{B1}, supporting the central dual-regression idea that the two branches are complementary rather than redundant.

The coordinate branch is comparatively stable, especially in EPE, but does not uniformly outperform fusion. At this stage, this is best treated as a descriptive observation about the current implementation rather than a final claim about the method more generally.

\subsection{Runtime Analysis}

Table~\ref{tab:latency} shows that on Jetson Orin Nano the 10-joint model is faster than the 21-joint baseline across all three prediction modes. The largest cost difference appears in fusion mode, where \texttt{B1} reduces end-to-end latency from 36.81 ms/image to 33.31 ms/image. The same trend is visible in forward prediction time. The reduction is modest rather than dramatic, which is consistent with the fact that much of the compute still resides in the shared backbone.

The gap between fusion and the individual branches is also informative. For \texttt{B0}, fusion adds roughly 3.2 ms/image over heatmap mode in the forward stage. Table~\ref{tab:latency_breakdown} shows that the non-model costs are also non-trivial: image read and preprocessing together account for more than 7 ms/image in the fused setting. This supports the repository's choice to benchmark the full pipeline rather than the model alone. These timings provide preliminary deployment information, but they are not yet direct measures of interaction quality in an application setting.

\subsection{Fusion Diagnostics}

Table~\ref{tab:fusion_diagnostics} provides additional evidence about how the heuristic selector behaves. The 21-joint model chooses the coordinate branch more often than the heatmap branch, whereas the 10-joint model is almost evenly split. The oracle-match rate remains well below 1.0 for both models, meaning that the present rule often disagrees with the lower-error branch when judged retrospectively on visible joints. The 10-joint model achieves a better oracle-match rate than the 21-joint baseline, but both leave substantial room for improvement.

These diagnostics show that the current fusion rule behaves differently across the two model settings and does not always match the retrospectively lower-error branch. This identifies a clear area for later refinement.

\section{CONCLUSIONS}

This initial draft summarizes the current RGB hand-pose pipeline implemented in the repository and presents the available accuracy and latency results in research-article form. The present findings show how the 21-joint and 10-joint variants behave on RHD, how the fusion output compares with heatmap-only and coordinate-only prediction, and how the current models perform under Jetson Orin Nano benchmarking. The draft therefore establishes a documented starting point for the project rather than a final account of the full investigation.

Several limitations remain. The current article reflects only the aggregated outputs already available in the repository and therefore emphasizes the RHD ablation study rather than cross-dataset transfer or task-based interaction evaluation. The full-frame resize preprocessing also means that detector quality is not part of the reported system behavior. In addition, the current fusion rule is hand-crafted and the oracle-match analysis shows that it is only partially aligned with the lower-error branch. Stereo input has not yet been integrated into the reported experiments, so the article does not yet address the full project scope described in the preliminary article. Future work can extend this draft by adding stereo-aware experiments, broader comparisons, application-level testing, and a more developed discussion of the results.
